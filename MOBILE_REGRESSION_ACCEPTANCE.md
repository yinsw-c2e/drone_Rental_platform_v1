# 移动端关键页面回归与截图验收标准

## 1. 目的

本清单服务阶段 `R10.02`，用于把当前移动端重构后的主链路页面，收成一份可重复执行的回归与截图验收标准。

验收重点不是“页面能打开”，而是确认以下四件事：

1. 页面对象边界是否正确
2. 角色入口是否正确
3. 状态、编号、来源标签是否一致
4. 布局是否存在截断、溢出、错位、入口断链

## 2. 验收环境

- 后端：`http://127.0.0.1:8080`
- 移动端 Web 预览：`http://127.0.0.1:3100`
- 推荐截图宽度：
  - 手机：`390 x 844`
  - Web 预览：Chrome 设备模式 `iPhone 14 Pro`
- 当前默认主链路已切到 `/api/v2`

## 2.1 当前阶段执行结论

基于 `2026-03-15` 的阶段 10 收口结果：

1. 移动端主链路页面已经切到 v2 业务对象口径
2. 本轮已通过：
   - `cd mobile && npx tsc --noEmit`
   - `cd mobile && npm run web:build`
3. 当前非阻塞项只有一个：
   - Web 构建仍有 `chunk > 500k` 告警，属于后续拆包优化项，不阻塞主链路回归
4. 本文档当前作为 `R10.02` 的正式回归基线，后续截图验收应以本清单为准

## 3. 截图统一标准

所有关键页面截图都至少满足以下要求：

1. 顶部标题完整可见，不被安全区或导航条裁切
2. 底部主操作区完整可见，不被 Tab 栏或安全区遮挡
3. 卡片圆角、阴影、边距一致，不出现半截卡片
4. 编号、状态、来源标签、主金额/主动作必须同时可见
5. 空状态页不能只剩一句“暂无数据”，必须有说明和建议动作
6. 错误状态、加载状态不应破坏整体布局
7. 同一业务对象在列表页与详情页的状态文案必须一致

## 4. 关键页面回归矩阵

### 4.1 首页驾驶舱

- 入口：一级 Tab `首页`
- 角色：综合 / 业主 / 机主 / 飞手
- 数据源：`GET /api/v2/home/dashboard`
- 必查项：
  - 角色切换不会显示不存在角色的入口
  - 驾驶舱卡片不截断
  - 紧急待办、进行中任务、市场动态按当前视图切换
  - 业主可见 `发布需求 / 浏览供给`
  - 机主可见 `查看新需求 / 发布供给`
  - 飞手可见 `待接派单 / 可报名需求 / 飞行记录`
- 截图要求：
  - 综合视图一张
  - 机主视图一张
  - 飞手视图一张

### 4.2 供给市场

- 入口：一级 Tab `市场` -> `供给市场`
- 角色：客户 / 综合 / 机主
- 数据源：`GET /api/v2/supplies`
- 必查项：
  - 只展示满足重载准入的供给
  - 卡片必须显示 `供给编号 / 标题 / 载重能力 / 定价方式 / 状态或来源标签`
  - 筛选项与平台语义一致：区域、场景、吊重能力、直达下单
  - 进入供给详情后对象一致
- 截图要求：
  - 列表页一张
  - 筛选展开一张

### 4.3 供给详情与直达下单确认

- 入口：供给市场卡片 -> 供给详情 -> 直达下单确认页
- 角色：客户
- 数据源：
  - `GET /api/v2/supplies/:supply_id`
  - `POST /api/v2/supplies/:supply_id/orders`
- 必查项：
  - 详情页不再展示旧租赁语义
  - 可见供给方、无人机、服务范围、定价方式
  - 下单确认页必须展示：
    - 起运地址
    - 送达地址
    - 服务时间
    - 重量/货物类型
    - 预估金额
  - 提交后返回的订单状态必须为 `pending_provider_confirmation`
- 截图要求：
  - 详情页一张
  - 下单确认页一张
  - 提交成功页或订单详情一张

### 4.4 需求市场

- 入口：一级 Tab `市场` -> `需求市场`
- 角色：机主 / 飞手 / 综合
- 数据源：
  - `GET /api/v2/owner/demands/recommended`
  - `GET /api/v2/pilot/candidate-demands`
- 必查项：
  - 页面表达的是“需求”，不是订单
  - 机主只看到 `报价`
  - 飞手只看到 `报名候选 / 取消候选`
  - 不再混入订单卡片、支付状态、订单动作
- 截图要求：
  - 机主视图一张
  - 飞手视图一张

### 4.5 需求详情与报价入口

- 入口：需求市场卡片 -> 需求详情 -> 报价页
- 角色：机主 / 飞手 / 客户本人
- 数据源：
  - `GET /api/v2/demands/:demand_id`
  - `POST /api/v2/demands/:demand_id/quotes`
  - `POST|DELETE /api/v2/demands/:demand_id/candidate`
- 必查项：
  - 客户本人可见报价列表入口
  - 机主可见 `提交报价`
  - 飞手可见 `报名候选/取消候选`
  - 列表页和详情页的需求编号、状态一致
- 截图要求：
  - 客户视角详情一张
  - 机主视角详情一张
  - 报价提交页一张

### 4.6 我的需求 / 我的报价 / 我的供给

- 入口：`我的`
- 角色：客户 / 机主
- 数据源：
  - `GET /api/v2/demands/my`
  - `GET /api/v2/owner/quotes`
  - `GET /api/v2/owner/supplies`
- 必查项：
  - 三个列表不混对象
  - 需求、报价、供给分别有自己的状态口径
  - 点击进入详情时对象、编号、状态不跳变
- 截图要求：
  - 三个列表各一张

### 4.7 订单列表与订单详情

- 入口：一级 Tab `履约`
- 角色：客户 / 机主 / 飞手 / 综合
- 数据源：
  - `GET /api/v2/orders`
  - `GET /api/v2/orders/:order_id`
- 必查项：
  - 列表只表达订单，不混需求、不混候选任务
  - 卡片必须显示：
    - 订单号
    - 当前状态
    - 来源标签
    - 金额
    - 时间
  - 详情页可见：
    - 来源信息
    - 承接方
    - 执行方
    - 执行模式
    - 派单摘要
    - 财务摘要
    - 时间线
- 截图要求：
  - 列表页一张
  - 详情页一张

### 4.8 正式派单列表与详情

- 入口：一级 Tab `履约` -> `正式派单`
- 角色：机主 / 飞手
- 数据源：
  - `GET /api/v2/dispatch-tasks`
  - `GET /api/v2/dispatch-tasks/:dispatch_id`
- 必查项：
  - 页面对象是“正式派单”，不是需求，也不是订单
  - 列表和详情状态一致
  - 飞手可以接受/拒绝
  - 机主可以重派
- 截图要求：
  - 列表页一张
  - 详情页一张

### 4.9 飞行监控

- 入口：
  - 订单详情 -> 飞行监控
  - 正式派单详情 -> 飞行监控
- 角色：客户 / 机主 / 飞手
- 数据源：`GET /api/v2/orders/:order_id/monitor`
- 必查项：
  - 两个入口都能进入同一监控页
  - 监控页同时显示：
    - 订单摘要
    - 当前正式派单
    - 最新位置
    - 活跃告警
    - 飞行统计
    - 时间线
- 截图要求：
  - 监控页一张

### 4.10 飞行记录

- 入口：飞手中心 -> 飞行记录
- 角色：飞手
- 数据源：`GET /api/v2/pilot/flight-records`
- 必查项：
  - 顶部统计来自真实履约飞行记录
  - 列表与统计口径一致
  - 不再出现“列表为空但统计巨大”的分裂
- 截图要求：
  - 列表+统计一张

### 4.11 我的页与角色档案

- 入口：一级 Tab `我的`
- 角色：所有角色
- 数据源：
  - `GET /api/v2/me`
  - `GET /api/v2/client/profile`
  - `GET /api/v2/owner/profile`
  - `GET /api/v2/pilot/profile`
- 必查项：
  - 不再出现“注册客户/成为客户”的重复动作
  - 角色能力以 `role_summary` 为准，不依赖旧 `user_type`
  - 机主、飞手入口按真实能力显示
  - 绑定飞手管理页双向可达
- 截图要求：
  - 我的首页一张
  - 机主档案一张
  - 飞手档案一张

- 首页驾驶舱

- 需求市场
- 订单列表
- 正式派单列表
- 飞行记录
