## 4. 核心判断

### 4.1 不能删除的业务环节

下面这些是真实业务需要，不能为了“看起来简单”直接删掉：

- 实名或基础身份校验
- 起终点、货物、时间窗口等最小成单信息
- 机主确认或报价确认
- 支付
- 派单或自执行决策
- 飞行执行留痕
- 交付确认

### 4.2 应该减少暴露的环节

下面这些流程可以保留在系统内部，但不应该要求大多数用户主动理解：

- `候选飞手池`
- `正式派单任务`
- `执行模式`
- `dispatch_bound_pilot / dispatch_pool` 等内部状态
- 飞行记录的底层采集细节

### 4.3 体验优化的正确方向

本轮不追求“业务上少一步”，而追求“用户少想一步、少做一步、少理解一层”。

具体做法固定为四种：

1. 后置
说明：先让用户发起，再补充详细资料

2. 并行
说明：系统审核、资质校验、内部准备尽量后台并行，不阻断首个动作

3. 折叠
说明：多个内部对象折叠成一个用户对象，例如统一折叠到 `订单进度`

4. 代办
说明：能由系统自动匹配、自动保存、自动推荐的，不再让用户手动串行操作

## 5. 双清单：真实业务步骤 vs 用户可感知步骤

## 5.1 客户链路

### 当前真实业务步骤

1. 注册/登录
2. 实名或客户能力校验
3. 录入运输信息
4. 选择路径：直达下单或发布需求
5. 等待机主确认或等待报价
6. 选择方案
7. 创建订单
8. 支付
9. 等待派单或机主自执行
10. 查看执行进度
11. 确认交付/收货

### 当前用户可感知步骤

用户通常会感知成下面这些动作：

1. 先理解自己该去“发需求”还是“看供给”
2. 先理解是否要升级企业
3. 先理解是否要做货物申报
4. 填一大段表单
5. 等报价
6. 选机主
7. 等确认
8. 去支付
9. 等派单
10. 再看执行

### 目标用户可感知步骤

标准化场景：

1. 输入核心运输信息
2. 看系统推荐的服务
3. 确认并提交订单
4. 跟进订单进度

非标场景：

1. 输入最小任务信息
2. 发布任务
3. 看报价并选方案
4. 支付
5. 跟进订单进度

## 5.2 机主链路

### 当前真实业务步骤

1. 创建机主档案
2. 添加无人机
3. 补充资质
4. 创建服务
5. 发布服务或参与报价
6. 确认订单
7. 自执行或发起派单

### 当前用户可感知步骤

1. 注册成为机主
2. 添加无人机
3. 理解多类资质
4. 逐个提交审核
5. 绑定飞手
6. 再发布服务
7. 才能开始接活

### 目标用户可感知步骤

1. 建立机主身份
2. 添加机型并创建服务草稿
3. 补齐关键资质
4. 上架服务或参与报价

## 5.3 飞手链路

### 当前真实业务步骤

1. 注册/登录
2. 实名
3. 建立飞手档案
4. 提交飞行资质
5. 获取任务资格
6. 接单
7. 执行飞行
8. 留痕与结算

### 当前用户可感知步骤

1. 先注册
2. 再实名
3. 再开通飞手身份
4. 再传飞行证
5. 再传无犯罪记录
6. 再传健康证明
7. 等审核
8. 再找任务

### 目标用户可感知步骤

1. 开通飞手身份
2. 补基础资质
3. 看可接任务
4. 接单并执行

补充说明：

- 更完整的资质继续保留
- 但改成分级准入，而不是全部前置阻断

## 6. 冻结后的前台对象模型

### 6.1 客户前台只保留 3 个核心对象

客户主要只需要理解：

- `服务`
- `任务`
- `订单进度`

客户不需要主动理解：

- `派单任务`
- `候选飞手`
- `执行模式`
- `飞行记录采集`

这些内部对象统一折叠到 `订单进度` 中，通过时间线、阶段标签和消息提醒表达

### 6.2 机主前台只保留 3 个核心对象

机主主要只需要理解：

- `机队`
- `待处理线索`
- `订单`

其中：

- `服务` 属于机队经营能力的一部分
- `线索` 包含公开任务、报价机会、待确认直达单
- `派单` 仍保留，但作为订单里的执行动作，而不是和订单同层竞争注意力

### 6.3 飞手前台只保留 3 个核心对象

飞手主要只需要理解：

- `可接任务`
- `执行中`
- `资质进度`

飞行记录保留，但属于“我的执行历史”，不应占据首页主入口

## 7. 冻结后的主入口策略

## 7.1 顶层 Tab 策略

维持当前 5 个一级入口，但职责固定：

- `首页`
- `市场`
- `进度`
- `消息`
- `我的`

职责定义：

- 首页：只回答“你现在最该做什么”
- 市场：只回答“你去哪里找服务或机会”
- 进度：只回答“已经开始的事情走到哪了”
- 消息：只回答“谁在联系你、系统提醒了什么”
- 我的：只放个人资料、资质和低频入口

## 7.2 客户首页固定双入口

客户首页主入口固定为：

1. `快速下单`
2. `发布任务`

解释规则：

- `快速下单` 面向标准化场景
- `发布任务` 面向非标或复杂场景

这两个入口必须高于“浏览市场”“查看对象分类”等平台视角入口

## 7.3 市场页默认角色偏好

- 客户默认进入 `看服务`
- 机主默认进入 `看需求`
- 飞手默认进入 `看需求` 或 `可接任务`

这样用户进入市场时，默认先看到与自己当前目标最相关的内容，而不是先自己切换语义

## 7.4 订单页成为唯一主跟进入口

一旦进入成交阶段，前台默认只推荐用户去 `订单进度` 跟进。

不再鼓励用户在成交后继续在这些页面来回切换：

- 市场页
- 派单任务列表
- 飞行记录列表

这些页面对普通客户应尽量退到次级入口

## 8. 目标旅程设计

## 8.1 客户标准化场景：快速下单

适用场景：

- 路线清楚
- 货物类型清楚
- 对时效要求明确
- 不想自己比多个报价

目标链路：

1. 首页点击 `快速下单`
2. 输入起点、终点、货物核心信息、时间
3. 系统推荐可用服务
4. 用户确认方案
5. 提交订单
6. 跟进订单进度

系统后台继续处理的真实流程：

- 机主确认
- 支付
- 派单/自执行
- 飞行执行
- 交付完成

但这些动作对用户应该主要呈现在一个订单时间线里，而不是拆成多个学习成本高的对象

降级路径：

当系统推荐不出可用服务时（供给不足、区域未覆盖、时间窗口冲突等），应：

1. 明确告知用户"当前暂无匹配服务"，避免空白页
2. 提供一键转为"发布任务"的入口，自动带入已填信息（起终点、货物、时间）
3. 用户无需重新填写，只需补充非标细节后即可发布
4. 发布成功后进入"等待报价"状态，与复杂场景链路汇合

## 8.2 客户复杂场景：发布任务

适用场景：

- 路线复杂
- 货物有特殊要求
- 需要多个机主报价比较

目标链路：

1. 首页点击 `发布任务`
2. 填最小任务信息
3. 发布成功
4. 后续逐步补充详细资料
5. 收到报价后选择方案
6. 跟进订单进度

关键原则：

- 第一屏先发起，不追求一次填完
- 附件、货物声明、特殊要求可以后补
- 报价比较视图承担“帮助决策”的工作，不再让用户自己来回翻详情

## 8.3 机主路径：先经营准备，再公开接活

目标链路：

1. 完成机主身份开通
2. 添加无人机
3. 创建服务草稿
4. 并行补资质
5. 资质达标后上架公开服务
6. 确认订单或参与报价

关键原则：

- 不要求先把所有资质全补完才能开始配置经营内容
- `服务草稿` 是缓冲层，允许机主先开始准备
- 资质是上架门槛，不是进入机主端的门槛

## 8.4 飞手路径：先接低门槛任务，再逐步解锁

目标链路：

1. 开通飞手身份
2. 完成基础资质
3. 查看可接任务
4. 接单执行
5. 继续补齐更高等级资质

关键原则：

- 用 `资质进度` 告诉飞手还差什么
- 用 `可接任务` 告诉飞手当前能做什么
- 不要求飞手先理解平台全部业务结构

## 10. 等待态预期管理数据来源

### 10.1 问题

N2.07 要求"每个状态都显示预计等待时间"，但这个时间从哪来需要明确，否则实现时会卡住。

### 10.2 数据来源策略（按优先级）

第一版（固定经验值）：

| 等待场景 | 预计时间 | 超时提示阈值 |
|---|---|---|
| 等待机主确认直达单 | ≤ 2 小时 | 4 小时 |
| 等待机主报价 | ≤ 24 小时 | 48 小时 |
| 等待派单分配 | ≤ 1 小时 | 2 小时 |
| 等待飞手接受派单 | ≤ 30 分钟 | 1 小时 |
| 等待飞行完成 | 取决于任务距离 | 预计飞行时间 × 1.5 |

后续迭代方向：

1. 基于历史数据动态计算：取同类订单 P50 响应时间作为预估值
2. 机主/飞手自填承诺时间：报价时附带预计响应时间，优先使用
3. 实时更新：结合当前队列深度动态调整

### 10.3 超时处理

当等待超过阈值时：

1. 向等待方发送"仍在等待"的状态提示，附带联系客服入口
2. 向责任方发送催促消息（通过系统消息写入会话）
3. 超过二级阈值（阈值 × 2）时，标记为"响应超时"并通知运营介入

## 11. 状态变更通知机制

### 11.1 问题

用户不应该依赖主动刷新来感知订单进度变化。虽然正式推送服务（极光/个推）暂缓，但系统内部的消息触达必须先建立。

### 11.2 本轮方案

关键状态变更应自动触发系统消息写入关联会话：

| 状态变更 | 通知对象 | 消息内容方向 |
|---|---|---|
| 订单创建 | 机主 | "您有一笔新订单待确认" |
| 机主确认 | 客户 | "机主已确认，请完成支付" |
| 支付成功 | 机主、飞手 | "客户已支付，订单进入执行准备" |
| 派单分配 | 飞手 | "您有一个新的飞行任务" |
| 飞手接受 | 机主、客户 | "飞手已接受任务，开始准备" |
| 飞行完成 | 客户 | "飞行已完成，请确认交付" |
| 订单取消 | 相关方 | "订单已取消" + 退款说明 |

### 11.3 实现要求

1. 系统消息写入复用 N1.03 的消息基础设施
2. 状态变更触发点统一在 service 层的状态推进方法中完成，不散落在 handler 中
3. 后续正式推送上线时，只需在系统消息写入后增加一个推送 hook，不需要改业务逻辑

## 13. 执行结论

本轮体验重设计的核心不是砍流程，而是：

1. 用户先做最短动作
2. 系统接住后续复杂流程
3. 内部对象退到后台
4. 前台只保留少量、稳定、可预测的对象和入口

如果后续实现过程中出现“这个流程业务上不能删”，默认先检查它是否可以：

- 后置
- 并行
- 折叠
- 代办

只有四种都不成立时，才允许继续保留为用户显式步骤。
