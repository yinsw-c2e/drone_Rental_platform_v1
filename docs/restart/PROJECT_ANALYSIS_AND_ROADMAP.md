## 一、项目总览

### 1.1 技术栈

| 层 | 技术 | 状态 |
|---|---|---|
| 后端 | Go 1.25 + Gin + GORM + MySQL 8 + Redis 7 | 稳定运行 |
| 移动端 | React Native 0.84 + React 19 + Redux Toolkit | 已重构至 v2 |
| 管理后台 | React 19 + Antd 6 + Vite | 基本可用 |
| 基础设施 | Docker Compose (MySQL + Redis)、GitHub Actions (Android APK) | 最低可用 |

### 1.2 业务定位

平台聚焦 **重载末端货物吊运**（起飞重量 ≥150kg、有效吊重 ≥50kg），服务四类角色：

- **客户**（默认角色）：发布需求、下单、付款
- **机主**：管理无人机资产、发布供给、报价、承接订单
- **飞手**：接派单、执行飞行、记录轨迹
- **复合身份**：同时持有多个角色能力

### 1.3 核心业务对象

`需求 → 报价/撮合 → 订单 → 派单任务 → 飞行记录 → 结算`

两条成单链路：
1. **需求市场转单**：客户发需求 → 机主报价 → 客户选机主 → 生成订单
2. **供给直达下单**：客户浏览供给 → 直接下单 → 机主确认 → 生成订单

### 1.4 重构完成度

根据 `REFACTOR_MASTER_TASKLIST.md`，**阶段 0 到阶段 10 共 60+ 个重构任务已全部标记完成 ✅**，v1 到 v2 的架构级重构在任务清单层面已经结项。

## 三、现存问题与欠缺（按严重程度排序）

### 🔴 P0 — 阻塞上线的硬伤

#### 3.1 V2 API 有 10 个端点处于 `NotImplemented` 状态

| 路径 | 说明 | 影响 |
|---|---|---|
| `POST /orders/:id/cancel` | 订单取消 | **客户无法取消订单** |
| `POST /orders/:id/start-preparing` | 开始准备 | 执行状态无法推进 |
| `POST /orders/:id/start-flight` | 开始飞行 | 执行状态无法推进 |
| `POST /orders/:id/confirm-delivery` | 确认交付 | 执行状态无法推进 |
| `GET /flight-records/:id` | 飞行记录详情 | 飞手看不到单次飞行详情 |
| `POST /flight-records/:id/positions` | 位置上报 | **无法实时监控** |
| `POST /flight-records/:id/alerts` | 告警上报 | 安全功能缺失 |
| `POST /flight-records/:id/complete` | 完成飞行 | 飞行记录无法关闭 |
| `GET /conversations` | 会话列表 | v2 消息体系不可用 |
| `GET /conversations/:id/messages` | 会话消息 | v2 消息体系不可用 |

**结论**：订单全生命周期的中间状态推进（准备→飞行→交付→取消）在 v2 实际断链。v1 有对应实现但已冻结写入。这是 **最严重的功能断裂**。

#### 3.2 飞行监控子系统在 V2 层面是空壳

v1 的飞行监控系统（位置上报、告警、电子围栏、轨迹录制、多点任务）功能完整，但 v2 只是占位。实际运行时仍需依赖 v1 接口，v2 无法独立支撑飞行过程的实时数据。

#### 3.3 消息与会话系统 V2 未实现

v2 路由中的 `conversations` 接口全部 `NotImplemented`，v1 有完整的 WebSocket + 会话 + 消息 + 未读计数实现，但 v2 没有衔接。

### 🟡 P2 — 影响用户体验和研发效率

#### 3.8 用户流程链条过长

**问题一：需求转单流程步骤过多（≥8 步）**

```
客户注册 → 客户升级(企业) → 货物申报 → 发布需求(草稿) → 发布需求(上线)
→ 等待机主报价 → 选择机主 → 生成订单 → 等待机主确认 → 支付 → 等待派单
→ 飞手接单 → 执行飞行 → 确认交付 → 确认收货 → 评价 → 结算
```

用户从"有运输需求"到"开始执行"至少经历 **10 个操作节点**。对比网约车（输入目的地 → 确认下单 → 等车来），心智负担极重。

**问题二：机主上架供给流程复杂**

```
注册 → 创建机主档案 → 添加无人机 → 无人机认证(适航证/UOM/保险) 
→ 绑定飞手(可选) → 发布供给 → 设定价格和服务区域
```

机主从"想接活"到"能被客户看到"至少需要 **6-7 步**，且每步都有审核等待。

**问题三：飞手入驻链条过长**

```
注册 → 实名认证 → 飞手注册 → 提交飞行证书 → 提交无犯罪记录 
→ 提交健康证明 → 等待审核 → 绑定无人机/接受机主邀请
```

飞手需要 **7-8 步 + 多轮审核等待** 才能开始接单。

#### 3.9 术语/文案理解门槛高

| 当前术语 | 用户困惑 | 建议优化 |
|---|---|---|
| 供给 (Supply) | "供给"是经济学概念，普通用户不理解 | → "服务" 或 "可用方案" |
| 需求市场 / 供给市场 | 双市场概念增加认知负荷 | → 统一为"找服务"/"发任务" |
| 派单任务 vs 订单 | 用户分不清两者关系 | → 订单内部嵌入执行状态，减少暴露 |
| 候选飞手池 | 内部术语暴露到前台 | → 对飞手："可报名的任务" |
| 履约 → 已改为"进度" | ✅ 已优化 | 继续保持 |
| 驾驶舱 | 首页曾叫"驾驶舱" | ✅ 已优化为内容导向 |

#### 3.10 管理后台功能缺失

- 无角色/权限管理系统（当前仅有 admin 单一角色）
- 无操作日志/审计追踪
- 无批量操作能力

- 无详情弹窗/抽屉（全部是列表页，缺少 CRUD 深度）

#### 3.11 安全机制不完善

## 四、用户旅程心智分析

### 4.1 客户视角 — 关键痛点

```
痛点地图：

[首页]                    ✅ 驾驶舱直觉化
    │
    ├─→ 发任务（需求）    ⚠️ 需要理解"需求"概念，填写字段 12+ 个
    │    └─→ 等报价       ❌ 被动等待，无预估时间
    │         └─→ 选机主   ⚠️ 如何判断报价合理性？缺少比较工具
    │              └─→ 订单确认 → 支付   ⚠️ 支付为 Mock
    │
    └─→ 找服务（供给市场）  ⚠️ "供给"概念需要解释
         └─→ 直达下单      ✅ 流程较短
              └─→ 等机主确认  ❌ 被动等待
```

**核心问题**：客户两条下单路径都存在"被动等待黑洞"——发需求等报价、直达下单等确认——但系统没有给客户任何预期管理（预计多久会有回复）。

### 4.2 机主视角 — 关键痛点

```
[首页]
    ├─→ 添加无人机 → 认证(4步) → 审核等待    ❌ 链条过长
    ├─→ 发布供给 → 等客户下单                  ⚠️ 被动
    ├─→ 浏览需求市场 → 报价                    ✅ 主动
    ├─→ 确认直达订单                            ✅ 简洁
    └─→ 发起派单 → 飞手接单                     ⚠️ 需理解派单概念
```

**核心问题**：无人机认证链条（适航证→UOM→保险→适航证书）是行业合规需要，但 **4 种认证串行审核** 的等待成本极高，应考虑并行审核 + 进度总览。

### 4.3 飞手视角 — 关键痛点

```
[首页]
    ├─→ 认证(6步) → 审核等待    ❌ 入驻门槛极高
    ├─→ 绑定机主/接受邀请        ✅ 流程合理
    ├─→ 报名候选需求              ✅ 主动
    ├─→ 接受派单 → 执行飞行       ⚠️ v2 飞行过程断链
    └─→ 飞行记录 → 结算           ⚠️ 结算流程不清晰
```

**核心问题**：飞手是平台供给侧的核心资源，但入驻门槛是三种角色中最高的。应考虑 **分级准入**（基础认证 → 先接简单任务 → 逐步补全资质）。

## 六、流程链条优化建议（详细设计）

### 6.1 推荐的简化方案：「一步到位」快速下单

当前最短链路（直达下单）：
```
浏览供给 → 选择服务 → 确认信息 → 等机主确认 → 支付 → 执行
```

建议增加「智能推荐下单」模式（面向首次用户）：
```
输入目的地和吊运内容 → 系统推荐最优方案 → 一键下单 → 支付 → 执行
```

实现方式：
- 后端增加 `POST /api/v2/quick-order` 接口
- 根据起点/终点/吊重自动匹配最优供给
- 前端在首页增加"快速下单"入口卡片
- 适用于标准化场景（如电网物资吊运），非标场景仍走需求市场

### 6.2 推荐的简化方案：「渐进式」角色认证

当前逻辑：所有认证完成后才能操作
建议改为：分级解锁

| 认证等级 | 要求 | 解锁能力 |
|---|---|---|
| L0 | 手机号注册 | 浏览供给、发布需求 |
| L1 | 实名认证 | 下单、支付 |
| L2 | 基础资质（飞行执照 / 企业营业执照） | 接单、报价 |
| L3 | 完整资质（UOM / 适航 / 保险 / 无犯罪记录） | 高价值订单、平台优先推荐 |

每个操作在不满足等级时弹出"还差 X 步即可解锁"的引导，而非全部前置阻断。

## 七、风险与注意事项

1. **v1/v2 共存的维护成本**：当前两套路由同时活跃，修改业务逻辑需要同步两处，应尽快推进 v1 下线。

2. **支付安全**：目前支付为 Mock，上线前必须完成真实支付通道的安全审计、回调验签、幂等性保障。

3. **数据一致性**：订单状态机在 v2 有断链（状态推进接口 NotImplemented），如果用户通过某些路径进入中间状态，可能导致订单"卡住"。

4. **合规风险**：平台涉及无人机飞行，需要确保空域报备等功能真正可用（当前有 v1 实现，需验证其合规性）。注：本平台为短距末端吊运，只需空域报备/临时空域申请，不涉及航线审批。

5. **`.env` 泄露**：`admin/.env` 已被提交到 Git，建议立即移除并加入 `.gitignore`，同时轮换可能已泄露的密钥。
