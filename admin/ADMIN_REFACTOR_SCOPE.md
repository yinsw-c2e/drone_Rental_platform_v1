# 后台管理端重构范围说明

## 1. 目的

本文件用于完成阶段 `R8.01`：把管理后台从旧的 `用户 / 无人机 / 订单 / 财务` 粗粒度结构，收敛到与新业务模型一致的对象管理视角。

它服务两个目标：

1. 明确后台下一阶段必须新增或改造的页面范围
2. 明确后台查询口径必须与新对象模型对齐，避免继续沿用旧混合语义

## 2. 当前后台现状

当前后台前端位于 `admin/src`，现有主菜单为：

- 数据概览
- 运营看板
- 智能报表
- 用户管理
- 飞手管理
- 客户管理
- 货物申报审核
- 无人机管理
- 订单管理
- 财务管理

当前问题：

- 没有独立的 `需求管理` 页面
- 没有独立的 `供给管理` 页面
- 没有独立的 `正式派单管理` 页面
- 没有独立的 `飞行记录管理` 页面
- 迁移审计与异常订单也没有承接页
- 订单列表仍主要沿用旧 `rental/cargo` 和旧状态口径

## 3. 目标后台对象模型

后台必须与移动端和后端 v2 模型保持一致，重点对象固定为：

- `users`
- `client_profiles`
- `owner_profiles`
- `pilot_profiles`
- `demands`
- `owner_supplies`
- `orders`
- `dispatch_tasks`
- `flight_records`
- `refunds / dispute_records / settlements`
- `migration_audit_records`

## 4. 后台目标菜单建议

建议后台按以下结构重排：

- 数据概览
- 用户与档案
  - 用户管理
  - 客户档案
  - 飞手档案
  - 机主档案 / 绑定飞手
- 市场对象
  - 需求管理
  - 供给管理
  - 报价与候选池（可作为详情侧栏先行）
- 履约对象
  - 订单管理
  - 正式派单管理
  - 飞行记录管理
- 财务与售后
  - 支付 / 退款 / 争议 / 结算
- 审核与运营
  - 无人机与资质审核
  - 货物申报审核
  - 迁移审计与异常看板

## 5. 本轮必须落地的后台页面

阶段 `R8.02` 最少需要落地以下 5 个对象页：

1. `需求管理`
2. `供给管理`
3. `订单管理`（按新字段补齐）
4. `正式派单管理`
5. `飞行记录管理`

每个页面至少要满足：

- 列表查询
- 关键状态筛选
- 详情查看
- 能看见对象编号、来源、参与方、状态

## 6. 各对象页的最小字段要求

### 6.1 需求管理

列表至少展示：

- 需求编号
- 标题
- 客户
- 场景类型
- 预算
- 状态
- 报价数
- 候选飞手数
- 创建时间

详情至少展示：

- 地址快照
- 货物信息
- 选中方案状态
- 转单结果

### 6.2 供给管理

列表至少展示：

- 供给编号
- 标题
- 机主
- 无人机
- 起飞重量
- 最大载重
- 计价方式
- 是否支持直达下单
- 状态

详情至少展示：

- 服务区域
- 时间窗
- 价格规则
- 关联无人机资质状态

### 6.3 订单管理

列表至少展示：

- 订单编号
- 订单来源 `demand_market / supply_direct`
- 客户
- 承接方
- 执行飞手
- 执行模式 `self_execute / dispatch_pool / bound_pilot`
- 是否待机主确认
- 是否待派单
- 状态
- 创建时间

详情至少展示：

- 来源对象
- 当前正式派单
- 支付 / 退款 / 结算摘要
- 时间线摘要

### 6.4 正式派单管理

列表至少展示：

- 派单编号
- 来源订单
- 目标飞手
- 派单来源
- 状态
- 发起时间
- 响应时间

详情至少展示：

- 候选来源
- 重派记录
- 拒绝原因 / 超时原因

### 6.5 飞行记录管理

列表至少展示：

- 飞行记录编号
- 订单编号
- 正式派单编号
- 飞手
- 起飞时间
- 结束时间
- 飞行距离
- 时长
- 最大高度
- 状态

详情至少展示：

- 轨迹统计
- 告警摘要
- 关联订单和派单

## 7. 后台接口适配原则

后台不应继续只依赖旧 `/api/v1/admin/orders` 一类混合口径接口。

本轮适配原则：

- 优先补“后台全局读取接口”，而不是复用“当前用户视角接口”
- 列表接口先满足只读管理需求
- 先把对象查清，再逐步补审批和运营动作

建议新增后台读取接口：

- `GET /api/v1/admin/demands`
- `GET /api/v1/admin/supplies`
- `GET /api/v1/admin/dispatch-tasks`
- `GET /api/v1/admin/flight-records`
- `GET /api/v1/admin/migration-audits`（阶段 `R8.03`）

## 8. 与阶段 9 的衔接

后台在阶段 8 的目标不是完成最终切流，而是先建立与新模型一致的运营视角。

阶段 9 再继续做：

- 双读校验
- 迁移审计看板
- 切换后台默认读路径
- 冻结旧混合语义写入
